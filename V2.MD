Breeze ASR Desktop V2.0 產品升級規格書
主題：新增影片檔抽取音軌後轉錄功能
1. 升級背景

目前系統已可處理音訊檔案，例如 mp3、wav，並完成本地語音辨識與字幕輸出。
但真實使用場景中，使用者提供的素材不一定是純音檔，而常常是以下形式：

手機錄影檔
線上會議錄影檔
螢幕錄影檔
課程錄影檔
監控或紀錄用途影片檔

這些檔案通常為 mp4，使用者真正需要的不是影片分析，而是：

直接把影片中的聲音抽出來，再交給既有 ASR 流程辨識。

如果系統無法直接支援影片檔，使用者就必須自行先轉檔，這會增加：

使用門檻
操作步驟
錯誤機率
對產品完整性的質疑

因此，V2.0 升級的核心目標是：

讓使用者可直接丟入影片檔，系統自動抽取音軌並完成轉錄，不要求使用者先做音訊轉檔。

2. 升級目標
2.1 核心目標

新增影片檔支援能力，讓系統可對影片容器中的音軌進行抽取與標準化，並沿用既有語音辨識流程完成輸出。

2.2 非目標

本次升級不包含以下範圍：

不做畫面分析
不做 OCR
不做影片內容辨識
不做人物、場景、事件理解
不做多模態推理
不解析影格
不支援影片剪輯

也就是說，本次升級的技術定位很明確：

V2.0 是影片抽音升級，不是影片 AI 分析升級。

3. 使用者價值
3.1 使用者可直接處理常見素材

使用者可直接將 mp4 拖入系統，不需要先用第三方工具轉成 mp3 或 wav。

3.2 降低使用門檻

把原本需要使用者自己理解的前置工作，收斂成產品內部自動處理。

3.3 提升產品完整度

從使用者視角來看，支援影片檔是語音轉文字工具應有的自然能力。
若產品只能吃音檔，會被感知為「半成品」。

3.4 保持既有核心不變

本次升級不改動 ASR 模型邏輯，只是在輸入層新增影片轉音訊能力，因此風險可控、投資報酬比高。

4. 升級範圍
4.1 新增支援格式

V2.0 新增：

mp4

後續保留擴充能力，但本版先不納入正式支援清單：

mov
mkv
avi

理由很務實：
mp4 已經覆蓋大多數使用者最常見的影片來源，先把主力路徑做穩，比一次擴到所有影片格式更合理。

4.2 輸入處理邏輯變更

原本輸入流程：

mp3 / wav → 音訊標準化 → ASR

V2.0 新流程：

mp3 / wav → 音訊標準化 → ASR
mp4 → 抽取音軌 → 音訊標準化 → ASR

也就是：

影片檔不直接進 ASR，而是先轉為標準音訊後再進入既有流程。

5. 功能需求
5.1 使用者功能需求
5.1.1 支援加入影片檔

使用者可透過以下方式加入 mp4：

拖曳到主畫面
手動選檔
加入整個資料夾時自動掃描
5.1.2 自動辨識檔案類型

系統需能判斷輸入檔案為：

音檔
影片檔

並走對應流程。

5.1.3 自動抽取音軌

若檔案為 mp4，系統自動：

讀取影片容器
擷取第一條可用音軌
轉為系統標準音訊格式
5.1.4 無需畫面解析

系統不得執行影格分析或影像處理，避免：

無謂效能消耗
架構複雜化
使用者誤解產品定位
5.1.5 輸出與既有音檔一致

由 mp4 抽音後轉錄的結果，輸出格式需與既有音檔流程一致：

.txt
_full.txt
.srt
6. 非功能需求
6.1 效能要求
抽音流程不得阻塞 GUI 主執行緒
抽音應在背景任務中進行
使用者需看得到抽音狀態與進度
6.2 穩定性要求
若影片檔無音軌，系統需可辨識並回報明確錯誤
若音軌解碼失敗，不得造成整體程式崩潰
單一檔案失敗時，不應中斷整批任務
6.3 可維護性要求
影片抽音功能必須與 ASR 推論核心解耦
不得把 mp4 特判邏輯硬塞入 transcriber 核心
6.4 可擴充性要求
後續新增 mov / mkv / avi 時，不應大幅修改主流程
未來若新增多音軌選擇，也應可在目前架構上擴充
7. 技術方案
7.1 推薦方案：ffmpeg 抽音

V2.0 建議採用：

ffmpeg 作為影片抽音核心工具

理由
支援度高
對影音容器處理成熟
對 mp4 音軌擷取非常穩定
不需要解析畫面即可抽音
後續擴充其他影片格式時仍具延展性
技術定位

ffmpeg 僅作為：

容器解析
音軌抽取
音訊格式轉換

不作為：

視訊分析工具
影像處理框架
7.2 不建議方案

以下方案不建議作為主實作：

7.2.1 OpenCV

原因：

你不需要畫面
引入它只會增加依賴與打包成本
7.2.2 直接把影片交給 ASR 模組處理

原因：

職責錯亂
讓 transcriber 汙染容器解析邏輯
未來維護成本會上升
7.2.3 依賴使用者自行安裝 ffmpeg

原因：

與本專案「降低環境依賴」目標衝突
會把部署問題再次丟回給使用者
8. 系統架構變更
8.1 原有架構
Input File
  └─ audio_loader
      └─ transcriber
          └─ exporter
8.2 V2.0 升級後架構
Input File
  └─ media_loader
      ├─ audio_loader
      └─ audio_extractor
           └─ temp wav
      └─ transcriber
          └─ exporter
8.3 新增模組

建議新增兩個核心模組：

media_loader.py

負責：

判斷輸入是音檔還是影片檔
分派到正確的處理流程
audio_extractor.py

負責：

從 mp4 抽取音軌
轉成標準暫存音訊
回傳暫存檔路徑或 waveform

9. 資料流程設計
9.1 音檔流程（既有）
mp3 / wav
  → load_audio()
  → resample to 16kHz mono
  → transcribe
  → export
9.2 影片流程（新增）
mp4
  → extract_audio_from_video()
  → temp wav
  → load_audio()
  → resample to 16kHz mono
  → transcribe
  → export
  → cleanup temp file
10. 暫存檔策略
10.1 必須有暫存區

因為影片抽音流程通常會先產生中繼音訊檔，所以必須設置：

runtime/temp/
10.2 暫存檔命名規則

建議使用：

原檔名
時間戳
隨機碼

例如：

input_audio_20260330_153012_ab12.wav

避免：

多工任務檔名衝突
同名影片重覆處理互蓋
10.3 清理規則
預設行為

任務完成後自動刪除暫存音檔。

例外情況

若使用者啟用「保留暫存音檔供除錯」，則保留。

異常中斷

程式下次啟動時可掃描 temp 資料夾，清理過期檔案。

11. GUI 規格變更
11.1 主畫面格式說明更新

原本顯示：

支援 mp3 / wav

V2.0 需更新為：

支援 mp3 / wav / mp4
MP4 僅抽取音訊，不解析畫面
11.2 任務清單新增欄位（可選）

建議任務列表可顯示：

輸入類型：Audio / Video
前處理狀態：待抽音 / 抽音完成 / 轉錄中
11.3 狀態顯示

當處理 mp4 時，使用者應看到明確狀態：

正在抽取音軌
正在轉換音訊格式
正在進行語音辨識

不要只顯示一個模糊的「處理中」。

11.4 設定頁新增選項

建議新增：

是否保留暫存抽音檔
影片抽音失敗時是否繼續下一檔
支援影片格式清單顯示（只讀說明）
12. 錯誤處理規格
12.1 無音軌

情境：

使用者丟入 mp4，但檔案沒有音訊

系統行為：

任務標記失敗
顯示：「此影片未包含可用音軌」
記錄 log
12.2 音軌抽取失敗

情境：

ffmpeg 無法處理該影片
檔案損毀
編碼異常

系統行為：

任務標記失敗
顯示：「音軌抽取失敗，請確認影片檔是否正常」
記錄錯誤細節到 log
12.3 抽取成功但後續轉錄失敗

情境：

抽出音訊沒問題，但 ASR 階段出錯

系統行為：

任務標記失敗
保留抽音結果供除錯（若設定允許）
GUI 顯示清楚失敗階段

13. 對既有系統的影響分析
13.1 正面影響
提升輸入彈性
降低使用者前置轉檔成本
提升產品完整性
強化會議錄影與錄課場景可用性
13.2 負面影響
打包體積可能增加
相依管理變複雜
前處理時間增加
失敗情境變多
13.3 結論

負面影響存在，但都屬於輸入前處理層的可控複雜度，不會對 ASR 核心造成致命污染。
因此此升級屬於：

高使用者價值、低至中等工程風險、值得投入的升級項目。

extract_success

ffmpeg 執行命令

抽音失敗原因

16. 使用者故事
User Story 1

作為一位一般使用者，
我希望可以直接把 mp4 拖進系統，
這樣我就不用自己先轉音檔。
